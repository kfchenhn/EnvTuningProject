# EnvTuningProject 渐进式迭代实现指南（基于自博弈 + AST + 双通道）

第一步：建立“可落地”的最小可运行骨架（你现在就可以开始）
- 目标：先不追求一次性全量重构，而是把“自博弈增强所需组件”接入现有 EnvTuning 架构。
- 已实现代码骨架：
  - `env_tuning/self_play/validators.py`：双维度结果验证器（状态维度 + 响应维度）。
  - `env_tuning/self_play/anchor_selector.py`：三级瀑布锚点选择器（批内同伴 > 历史经验 > 课程诱导）。
  - `env_tuning/self_play/ast_diagnostics.py`：AST风格结构标准化 + 首个逻辑分歧定位。
  - `env_tuning/self_play/pipeline.py`：自博弈协调器，生成反事实偏好样本与 GRPO 优化提示。
- 你要做的事：
  1. 保持现有 `new_multi_turn_fc.py` 交互流程不变，先在离线分析环节调用上述模块。
  2. 从训练日志导出“成功轨迹/失败轨迹”样本，验证诊断器是否能稳定找出第一逻辑分歧点。
  3. 不直接改训练主循环，先做离线原型验证，避免影响已有 Stage1~3 训练稳定性。

第二步：在评分系统中引入“结果驱动双维度验证”
- 目标：将奖励信号从“纯格式/单一成功率”升级为“状态一致性 × 响应有效性”。
- 已完成接入：
  - `env_tuning/interaction/score_calculator.py` 已改为调用 `DualOutcomeValidator`，输出二元乘积奖励。
- 你要做的事：
  1. 对比改造前后同一批验证集样本的分布（奖励均值、方差、极端值比例）。
  2. 检查是否出现“状态正确但响应缺失”或“响应看似正确但状态不一致”的分裂样本。
  3. 把这些分裂样本打标签，作为后续自博弈诊断优先数据源。

第三步：接入“锚点选择队列”，解决冷启动失败
- 目标：保证失败轨迹总能找到可比较的参考锚点。
- 核心策略（按优先级）：
  1. 批内同伴锚点（同一任务、同批采样中 reward 更高者）。
  2. 历史自我锚点（经验池中同 task signature 的最佳成功轨迹）。
  3. 课程诱导锚点（人工/规则引导跑通后写入经验池）。
- 对应实现：`AnchorSelector` + `AnchorMemory`。
- 你要做的事：
  1. 在每个 rollout batch 结束后，构造 `TrajectoryCandidate` 并写入 memory。
  2. 记录“锚点来源占比”（peer/history/curriculum）作为训练健康度指标。
  3. 若 curriculum 占比长期 >40%，说明模型仍过度依赖外部搀扶，需要延长阶段二或降低任务难度。

第四步：部署 AST 后见诊断，定位第一逻辑分歧
- 目标：把“序列级失败”变成“动作级可归因失败”。
- 已实现能力：
  - `normalize_trajectory(...)`：去噪字段、参数字典序、并行调用稳定排序。
  - `find_first_logic_divergence(...)`：识别 strategy divergence / parameter divergence。
- 你要做的事：
  1. 将模型的工具调用日志转换成统一结构（每轮 action list）。
  2. 对每个失败样本匹配 anchor 后运行 divergence 检测。
  3. 保存诊断元数据（turn_index, action_index, divergence_type）到训练样本。

第五步：构建通道B（在线纠错重试）
- 目标：在在线交互中复用环境提示系统，强制挽救失败轨迹，产出高价值锚点。
- 具体做法：
  1. 在执行报错时，触发增强生成器（你的现有 env tuning 提示模块）。
  2. 通过宪法裁判（安全/泄漏/可操作性）过滤提示后注入上下文。
  3. 若重试成功，将“提示前失败轨迹 + 提示后成功轨迹”作为黄金对比数据。
- 实操建议：
  - 先把通道B仅用于 Stage2/3；Stage4 默认关闭，保持真实部署一致性。

第六步：构建通道A（离线反事实训练 + GRPO）
- 目标：把外部提示转化为参数内化，避免“拐杖依赖”。
- 已实现基础：`SelfPlayCoordinator.build_grpo_advantage_hints(...)` 输出关键节点正负样本位置信息。
- 你要做的事：
  1. 在 GRPO 数据预处理阶段注入节点级别优势：
     - 第一逻辑分歧点的成功动作 -> 正优势。
     - 同点失败动作 -> 负优势。
  2. 保持你在方案中设定的稳定性超参：
     - `clip_ratio_low=0.2`
     - `clip_ratio_high=0.28`
     - `kl_loss_coef≈0.1`
  3. 监控 KL、熵、梯度范数，若熵快速塌缩，优先提高 KL 或降低学习率。

第七步：落地四阶段课程（渐进式）
- 阶段一：语法与格式重塑
  - 只训格式、工具名、调用结构；关掉复杂提示和复杂任务。
- 阶段二：真值引导冷启动
  - 开启细粒度奖励 + 强提示辅助，确保基础轨迹能跑通。
- 阶段三：自博弈与逻辑内化
  - 开启锚点队列 + AST 诊断 + 通道A/B。
  - 动态衰减通道B提示强度。
- 阶段四：鲁棒性与实战演练
  - 关闭提示注入，加入噪声与冗余步惩罚，强化 OOD 泛化。

第八步：建议的迭代节奏（每周可执行）
- 迭代1（1周）：完成离线诊断评估（不改主训练）。
- 迭代2（1周）：将锚点队列接入 rollout 后处理，观察锚点来源分布。
- 迭代3（1~2周）：接入通道B在线重试，验证成功率提升与数据质量。
- 迭代4（1~2周）：把反事实样本喂给 GRPO（通道A），调稳定性超参。
- 迭代5（持续）：按四阶段课程做自动调度，做 OOD 压测。

第九步：验收标准（建议）
- 训练稳定性：
  - 梯度范数无持续爆炸；KL 和熵维持在可控区间。
- 数据质量：
  - 失败样本中可定位第一逻辑分歧点比例 > 80%。
  - 有效锚点覆盖率（peer+history）逐步提升。
- 能力提升：
  - BFCL 多轮任务成功率阶段性增长。
  - OOD 任务（ACEBench 类）相对基线有明显增益。

第十步：你下一步最该先做什么
- 立即从你当前训练日志抽 100 个失败样本：
  1. 用 `AnchorSelector` 找锚点；
  2. 用 `find_first_logic_divergence` 做分歧定位；
  3. 用 `SelfPlayCoordinator` 生成反事实样本；
  4. 统计“可诊断率 + 参数分歧/策略分歧占比”。
- 这一步不需要改训练主循环，但能最快验证你方案是否在工程上闭环。
